1. Analízis modell

1. Bevezetés

Ez a dokumentum a rendszer analízis modelljét tartalmazza, amely az SRS (Szoftver Követelmény Specifikáció) alapján készült. A modell célja, hogy a rendszer statikus, dinamikus és funkcionális szerkezetét írja le, megalapozva ezzel a részletes tervezést és implementációt.

2. Statikus modell: Kezdeti Osztálydiagram

Az SRS elemzése során a következő kulcsfontosságú osztályokat azonosítottuk:

  • Felhasználó (User): Absztrakt osztály, amely a rendszer összes felhasználójának közös tulajdonságait (név, email, jelszó) és viselkedését (bejelentkezés, regisztráció) foglalja össze.
  • Oktató (Instructor): A Felhasználó osztályból származik. Felelős kurzusok létrehozásáért, feladatok kiírásáért és értékelésért.
  • Diák (Student): A Felhasználó osztályból származik. Képes kurzusokhoz csatlakozni és feladatokat beadni.
  • Kurzus (Course): Az oktatás központi egysége. Tartalmazza a bejegyzéseket, és kezeli a hozzá tartozó diákokat és oktatókat.
  • Bejegyzés (Post): Absztrakt ősosztály a kurzus falán megjelenő elemek számára.
  • Közlemény (Announcement): A Bejegyzés egy konkrét típusa, egyszerű tájékoztató szöveget tartalmaz.
  • Feladat (Assignment): A Bejegyzés egy speciális típusa, amelyhez határidő és beadási lehetőség tartozik.
  • Beadás (Submission): Egy konkrét Feladat-hoz tartozó, egy Diák által feltöltött megoldás. Tartalmazhat szöveget vagy fájlokat.
  • Értékelés (Grade): Egy Beadás-hoz rendelt pontszám és/vagy szöveges visszajelzés, amelyet az Oktató hoz létre.

Főbb kapcsolatok (Asszociációk):

  • Egy Oktató több Kurzus-t is létrehozhat (1-N).
  • Egy Kurzus-hoz több Diák is tartozhat (1-N). A diákok és kurzusok között sok-a-sok kapcsolat van, amit egy kapcsolóosztály kezel.
  • Egy Kurzus több Bejegyzés-t tartalmaz (1-N, kompozíció).
  • Egy Feladat-hoz több Beadás is tartozhat (1-N).
  • Egy Diák egy Feladat-hoz egy Beadás-t küldhet be (1-1).
  • Egy Beadás-hoz egy Értékelés tartozik (1-1).

2.1. Alrendszerek

A rendszert a jobb átláthatóság érdekében a következő funkcionális alrendszerekre bontottuk:

  1. Felhasználókezelési alrendszer: Felelős a felhasználók regisztrációjáért, hitelesítéséért és jogosultságkezeléséért. Osztályok: Felhasználó, Oktató, Diák.
  2. Kurzusmenedzsment alrendszer: Kezeli a kurzusok létrehozását, a diákok csatlakozását, valamint a kurzus falán megjelenő bejegyzéseket. Osztályok: Kurzus, Bejegyzés, Közlemény, Feladat.
  3. Beadás- és Értékeléskezelő alrendszer: Felelős a feladatok beadási és értékelési folyamatáért. Osztályok: Beadás, Értékelés.

2. Rendszertervezés

1. Bevezetés

Ez a dokumentum a rendszer magas szintű technikai terveit tartalmazza, beleértve a felhasználói felület vázlatos felépítését és az adatbázis-architektúrát.

2. Felhasználói felület

A felhasználói felület (UI) modern, letisztult és reszponzív lesz. A főbb képernyők tervei:

  • Vezérlőpult (Dashboard): Bejelentkezés utáni kezdőképernyő. Kártya alapú elrendezésben jeleníti meg a felhasználó kurzusait. Egy oldalsávban a közelgő határidők és a legutóbbi értesítések láthatók.
  • Kurzus oldal: Kétoszlopos elrendezés. A fő (szélesebb) oszlop a kurzus "fala", ahol időrendben jelennek meg a közlemények és feladatok. A keskenyebb oszlopban a kurzus adatai és a közelgő határidők listája található.
  • Feladat oldal: Részletes nézet egy kiválasztott feladatról. Tartalmazza a leírást, a csatolt fájlokat. Diákok számára itt található a beadási felület, oktatók számára pedig a beérkezett munkák listája.

3. Adatmodellek

3.1. Adatbázis kezelő kiválasztása Az alkalmazás adatainak perzisztens tárolására a PostgreSQL relációs adatbázis-kezelőt választottuk. Indokok: nyílt forráskódú, megbízható, robusztus, jól skálázható és széles körben támogatott a Node.js környezetben (pl. Sequelize, TypeORM ORM-eken keresztül).

3.2. Szemantikai adatmodell (ER-diagram leírása)

  • Egyedek: Users, Courses, Posts, Submissions, Grades.
  • Kapcsolatok:
    • Users és Courses között sok-a-sok kapcsolat van egy CourseMemberships kapcsolótáblán keresztül, ami tárolja a felhasználó szerepkörét (oktató/diák) az adott kurzusban.
    • Courses és Posts között egy-a-sok kapcsolat van (egy kurzusnak több bejegyzése lehet).
    • Users (specifikusan Diákok) és Posts (specifikusan Feladatok) között sok-a-sok kapcsolat van a Submissions táblán keresztül.
    • Submissions és Grades között egy-az-egyhez kapcsolat van.

3.3. Relációs adatmodell (Táblák vázlata)

CREATE TABLE Users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE Courses (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    course_code VARCHAR(10) UNIQUE NOT NULL,
    creator_id INTEGER REFERENCES Users(id)
);

CREATE TABLE CourseMemberships (
    user_id INTEGER REFERENCES Users(id),
    course_id INTEGER REFERENCES Courses(id),
    role VARCHAR(50) NOT NULL, -- 'instructor' or 'student'
    PRIMARY KEY (user_id, course_id)
);

CREATE TABLE Posts (
    id SERIAL PRIMARY KEY,
    course_id INTEGER REFERENCES Courses(id),
    author_id INTEGER REFERENCES Users(id),
    title VARCHAR(255) NOT NULL,
    content TEXT,
    post_type VARCHAR(50) NOT NULL, -- 'announcement' or 'assignment'
    due_date TIMESTAMP WITH TIME ZONE, -- NULL for announcements
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE Submissions (
    id SERIAL PRIMARY KEY,
    assignment_id INTEGER REFERENCES Posts(id),
    student_id INTEGER REFERENCES Users(id),
    submission_date TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    content TEXT
    -- Fájlkezeléshez egy külön tábla is szükséges lehet.
);

3.4. Az adatbázis kezelővel kapcsolatot tartó osztályok Az alkalmazás a Repository mintát fogja használni az adatbázis-műveletek elvégzésére. Minden fő entitáshoz (pl. User, Course) tartozni fog egy repository osztály (pl. UserRepository, CourseRepository), amely elrejti az adatbázis-specifikus logikát a szolgáltatási réteg elől. Például: UserRepository operációi: findById(id), findByEmail(email), save(user).

3. Szótár

Ez a dokumentum a projektben használt kulcsfontosságú fogalmakat és azok definícióit tartalmazza az egységes értelmezés érdekében.

  • Aktor (Actor): A rendszert használó személy vagy külső rendszer, amely interakcióba lép a szoftverrel. A projektben két fő aktor van: az Oktató és a Diák.

  • Bejegyzés (Post): A kurzus falán megjelenő bármilyen tartalmi elem. Két fő típusa van: a Közlemény és a Feladat.

  • Diák (Student): A rendszer azon felhasználója, aki kurzusokhoz csatlakozik, hozzáfér a tananyagokhoz, és feladatokat ad be.

  • Értékelés (Grade): Az a folyamat, amely során az Oktató pontszámmal és/vagy szöveges visszajelzéssel látja el a Diák által beadott feladatot.

  • Feladat (Assignment): Egy speciális típusú Bejegyzés, amelyhez beadási határidő és értékelési lehetőség tartozik. A diákoknak erre kell megoldást beküldeniük.

  • JWT (JSON Web Token): Egy iparági szabvány (RFC 7519) a felek közötti információ biztonságos, kompakt és önálló átvitelére. A projektben a felhasználói authentikáció és autorizáció megvalósítására szolgál.

  • Kurzus (Course): Az oktatás központi digitális egysége, amely egy adott tantárgyhoz vagy témakörhöz kapcsolódik. Az Oktatók ezen keresztül szervezik a tananyagot, osztják ki a feladatokat és kommunikálnak a Diákokkal.

  • Kurzuskód (Course Code): Egy egyedi, véletlenszerűen generált karaktersorozat, amelyet a kurzust létrehozó Oktató oszt meg a diákokkal. Ennek segítségével tudnak a Diákok csatlakozni a kurzushoz.

  • Közlemény (Announcement): Egy általános célú Bejegyzés a kurzus falán, amely tájékoztató jellegű információkat tartalmaz (pl. emlékeztetők, segédanyagok linkjei). Nem igényel beadandót a diákoktól.

  • Oktató (Instructor): A rendszer adminisztratív jogosultságokkal rendelkező felhasználója, aki kurzusokat hoz létre és kezel, feladatokat ír ki, és értékeli a beérkezett diák munkákat.

  • RESTful API (Representational State Transfer Application Programming Interface): Egy architekturális stílus szoftverrendszerek hálózaton keresztüli kommunikációjára. A projektben a kliensoldali (frontend) és a szerveroldali (backend) alkalmazás közötti adatcserét definiálja.

  • Vezérlőpult (Dashboard): Az a kezdőoldal, amelyet a felhasználó a bejelentkezés után lát. Itt áttekintheti a kurzusait, a közelgő határidőket és a legutóbbi értesítéseket.

  • Alrendszer (Subsystem): A szoftverrendszer funkcionálisan összetartozó részegysége, amely meghatározott feladatokat lát el. Például: Felhasználókezelési alrendszer.

  • ER-diagram (Entity-Relationship Diagram): Egy adatbázis szemantikai modelljének grafikus reprezentációja, amely az egyedeket, azok tulajdonságait és a közöttük lévő kapcsolatokat ábrázolja.

  • Objektum (Object): Egy osztály konkrét példánya a memóriában, amely állapottal (attribútumok) és viselkedéssel (operációk) rendelkezik.

  • Osztálydiagram (Class Diagram): Az UML statikus modellező eszköze, amely a rendszer osztályait, azok attribútumait, operációit és a közöttük lévő kapcsolatokat (pl. öröklődés, asszociáció) mutatja be.

  • Perzisztencia (Persistence): Az adatok megőrzésének képessége a program futásának befejezése után is, tipikusan adatbázisban vagy fájlrendszerben történő tárolással.

  • Relációs adatmodell (Relational Data Model): Adatok táblázatokban (relációkban) való szervezése, ahol a sorok a rekordokat, az oszlopok pedig az attribútumokat képviselik.

  • Szekvenciadiagram (Sequence Diagram): Az UML dinamikus modellező eszköze, amely az objektumok közötti üzenetváltásokat időbeli sorrendben ábrázolja egy adott use case vagy funkció végrehajtása során.
